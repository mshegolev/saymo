# Milestone v1.2: Trigger Training Console

**Status:** Shipped 2026-05-20
**Archived:** 2026-05-20
**Phases:** 8-10
**Total Plans:** 11

## Overview

Milestone v1.2 turned recorded call samples into a practical local training
loop for named session review, relabeling, readiness checks, and safe
classifier promotion. The shipped workflow keeps audio and metadata local,
lets the user correct labels without editing JSON manually, and only allows
learned behavior to assist live calls after evidence-backed profile gates pass.

## Phases

### Phase 8: Capture Session Ledger

**Goal:** Treat each recording run as a named local session with auditable
counts and metadata.
**Depends on:** v1.1 trigger sample metadata and capture command
**Requirements:** SES-01, SES-02, SES-03

Plans:
- [x] 08-01: Add session metadata and storage helpers
- [x] 08-02: Wire session ids into trigger-capture
- [x] 08-03: Add session list/summary commands and docs

Delivered:
- `trigger-capture --session` stores a generated session id, name, and sequence
  on every saved sample while preserving legacy samples.
- Session ledgers capture start/finish state, skipped-silence counts, category
  counts, speaker counts, answer-decision counts, and readiness labels.
- `trigger-sessions list` and `trigger-sessions summary` inspect prior local
  recording runs by profile and session id/prefix.

### Phase 9: Review And Relabel Workflow

**Goal:** Make saved samples easy to triage and correct from the CLI after a
call.
**Depends on:** Phase 8
**Requirements:** REV-01, REV-02, REV-03, REV-04

Plans:
- [x] 09-01: Extend sample filtering and disagreement detection
- [x] 09-02: Add category relabel command
- [x] 09-03: Add interactive review queue
- [x] 09-04: Add session-aware sanitized reports and docs

Delivered:
- `trigger-samples list` filters by profile, session, category, speaker,
  answer decision, classifier disagreement, and date range.
- `trigger-samples category` corrects sample categories and moves the JSON/WAV
  pair when possible.
- `trigger-samples review` replays a filtered queue and accepts category,
  speaker, answer-decision, skip, and quit actions.
- `trigger-samples report` exports sanitized session/category summaries without
  raw audio payloads or transcript text.

### Phase 10: Classifier Readiness Gate

**Goal:** Let learned behavior assist live calls only when local evidence is
strong enough and deterministic safety checks still pass.
**Depends on:** Phase 9
**Requirements:** CLS-01, CLS-02, CLS-03, CLS-04

Plans:
- [x] 10-01: Add classifier readiness metrics
- [x] 10-02: Add holdout evaluation reporting
- [x] 10-03: Add guarded per-profile live-assist config
- [x] 10-04: Add live-assist diagnostics and docs

Delivered:
- `trigger-classifier readiness` reports accepted/rejected label balance,
  category coverage, mention-vs-handoff coverage, and threshold failures.
- `trigger-classifier evaluate` reports local holdout confusion matrix,
  precision, recall, accuracy, and split counts.
- `trigger-classifier live-assist status|enable|disable` stores guarded
  per-profile opt-in state and refuses stale or insufficient models.
- `trigger-check --live-assist` explains enabled state, model status,
  confidence, final action, and reason locally.

## Milestone Summary

Key decisions:
- Treat capture runs as first-class sessions so samples can be reviewed in the
  context of a specific meeting.
- Keep review/relabel operations local and CLI-first so training data can be
  corrected without hand-editing metadata.
- Require balanced evidence and model fingerprint checks before live assist can
  be enabled for a profile.
- Keep deterministic trigger/addressing checks as the live-call safety boundary;
  live assist can only downgrade an answer candidate.

Issues resolved:
- Recorded training samples can now be grouped by meeting/session.
- Misclassified samples can be filtered, replayed, and corrected in bulk.
- Mention-only samples and handoff/request samples are checked separately for
  readiness.
- Live classifier assist now has explicit readiness, evaluation, and diagnostic
  commands instead of hidden runtime behavior.

Deferred:
- Fully local diarization as a first-class Saymo component.
- Provider-specific UI regression checks when call-provider web apps change.
- Voice-clone model training from call recordings, which remains out of scope
  for trigger-decision training.

Technical debt:
- Classifier quality still depends on collecting enough real accepted/rejected
  samples under `~/.saymo/trigger_samples/`.
- Provider automation robustness remains tied to Chrome call-provider DOM/API
  stability until a future UI regression milestone.
