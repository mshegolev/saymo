# Milestone v1.1: Call Intelligence Loop

**Status:** Shipped 2026-05-18
**Archived:** 2026-05-19
**Phases:** 5-7
**Total Plans:** 9

## Overview

Milestone v1.1 builds on the completed Speedly Catcher/Sayer work. The focus
was Advanced Call Intelligence: captured samples should explain speaker
context, learned trigger confidence, and provider-specific latency without
requiring cloud services or risky live-call behavior changes.

## Phases

### Phase 5: Speaker-Aware Sample Loop

**Goal:** Let the user label who spoke in captured windows and evaluate trigger
behavior separately for `me`, `other`, and `unknown` speakers.
**Depends on:** v1.0 sample review workflow
**Requirements:** SPK-01, SPK-02, SPK-03

Plans:
- [x] 05-01: Add speaker label metadata support
- [x] 05-02: Make trigger evaluation speaker-aware
- [x] 05-03: Add sample label review commands and docs

Delivered:
- New trigger-capture samples store `speaker`, defaulting to `unknown`.
- Legacy samples without labels remain valid as `unknown`.
- `trigger-eval` reports speaker-grouped records, misses, false positives, and
  answer counts.
- `trigger-samples label` updates speaker metadata locally.

### Phase 6: Local Trigger Classifier

**Goal:** Train a local sample-based classifier from accepted/rejected trigger
decisions and compare it against deterministic gating before it affects calls.
**Depends on:** Phase 5
**Requirements:** CLF-01, CLF-02, CLF-03

Plans:
- [x] 06-01: Add accepted/rejected sample labels
- [x] 06-02: Train lightweight local trigger classifier
- [x] 06-03: Add shadow-mode classifier diagnostics

Delivered:
- Sample metadata supports `answer_decision` with accepted, rejected, and
  unlabeled states.
- `trigger-classifier train|inspect|delete` manages local JSON classifier
  artifacts.
- Training refuses insufficient accepted/rejected labels.
- `trigger-eval` and `trigger-check` can show classifier shadow confidence
  without changing live answer decisions.
- Post-phase refinement separates `mentioned_me` samples from true
  `asked_to_speak` handoffs, reducing false positives when the user's name is
  only mentioned.

### Phase 7: Provider Latency Probe

**Goal:** Measure end-to-end latency for active Chrome call providers so the
user can distinguish audio/STT delays from provider mute/playback delays.
**Depends on:** Phase 6
**Requirements:** LAT-01, LAT-02, LAT-03

Plans:
- [x] 07-01: Add provider latency probe command
- [x] 07-02: Capture segmented provider/playback timings
- [x] 07-03: Add latency history export and docs

Delivered:
- `provider-latency` runs one active-call probe for a configured profile.
- Reports include capture, transcription, trigger decision, provider unmute,
  playback start, playback duration, and mute recovery.
- Blocked provider steps report explicit reasons.
- Probe history is written as local JSON and Markdown by profile/provider.

## Milestone Summary

Key decisions:
- Use local speaker-label sidecars before requiring any diarization engine.
- Keep the learned trigger classifier in shadow mode before it can affect live
  calls.
- Keep mention-vs-handoff classification deterministic until enough real
  samples justify learned live behavior.
- Measure provider latency through existing Chrome provider abstractions.

Issues resolved:
- Captured samples now explain speaker context.
- Accepted/rejected trigger decisions can train a local classifier.
- Name mentions such as "working with Misha" are separated from floor handoffs
  such as "word to Misha" in saved sample categories.
- Provider mute/playback latency can be measured separately from catch-path
  latency.

Deferred:
- Required local diarization model installation.
- Enabling the trained classifier in live auto-mode.
- Provider UI regression checks.

Technical debt:
- Classifier quality, provider history, and mention/handoff heuristics require
  real local sample data from actual calls before they are useful as tuning
  inputs.
